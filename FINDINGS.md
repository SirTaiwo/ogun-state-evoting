# Findings — Ogun State E-Voting Case Study

This document records observations, bugs discovered, and lessons learned during the study and adaptation of an open-source e-voting reference implementation for the Ogun State case study.

## 1. Authorship & Context

The base codebase (`server.js`, `db.js`, `seed.js`, `*.html`, `styles.css`) was generated by an AI coding tool (Cowork) using a single prompt, and made available as a reference implementation at https://github.com/SirTaiwo/electronic-voting.

This case study repository (`ogun-state-evoting`) builds on that base by:

- Adapting seed data to Ogun State context (LGAs, candidates, voters)
- Adding new database tables and API endpoints (LGA reference, validation)
- Modifying the frontend to use a validated LGA dropdown
- Fixing bugs discovered while studying the code
- Documenting findings and security observations (this document)

I am the author of all adaptations, fixes, and documentation in this repository. I am not the original author of the cryptographic architecture or initial system design.

The system is for **academic and civic-research purposes only**. It is not intended for use in any real election.

## 2. Bugs Discovered

Three notable issues surfaced during the study. Two have been fixed in the upstream reference repo or this case study; the third is documented for future work.

### Bug 1: Content Security Policy blocked inline event handlers

**Symptom:** Vote buttons on the voter page silently failed to respond. Clicking them produced no visible feedback. The server received no request because the click handler never executed.

**Diagnosis:** The browser DevTools Console revealed a CSP violation: "Refused to execute inline event handler because it violates the following Content Security Policy directive: 'script-src-attr 'none''."

The Helmet security middleware was correctly configured to allow inline `<script>` tags (`scriptSrc: ['self', 'unsafe-inline']`) but did not explicitly set `scriptSrcAttr`, which controls inline event handlers like `onclick="..."`. Helmet's default for unset directives is `'none'`, meaning all inline handlers were blocked.

This was an internally inconsistent configuration: the HTML pages depended on inline handlers, but the CSP rejected them.

**Fix:** Added `scriptSrcAttr: ["'self'", "'unsafe-inline'"]` to the CSP directives in `server.js`.

**Reference:** Commit `0bf46aa` in the upstream Electrovote repo.

**Future improvement:** A cleaner long-term fix is to refactor the HTML to use `addEventListener` instead of inline `onclick`, which would let the stricter CSP remain in place.

### Bug 2: Receipt code not displayed to voter after voting

**Symptom:** After casting a vote, the voter page displays "You have voted in this election" and shows a receipt-verification widget — but the voter is never shown their own receipt code. The receipt is generated and stored in the database, but never rendered in the UI.

**Diagnosis:** This is a UI flow gap rather than a logic bug. The server correctly returns the receipt code in its JSON response after a successful vote. The frontend code does not extract it from the response and display it to the voter.

Without the receipt code, the voter has no way to independently verify their own vote — which is exactly the property the cryptographic receipt system is designed to provide.

**Status:** Documented; not yet fixed in this repository. Receipt code can be retrieved by direct database query during testing.

**Future improvement:** Modify the voter page JavaScript to display the receipt code in a prominent box immediately after voting, with a copy-to-clipboard button.

### Bug 3: Server crashes silently leave zombie processes

**Symptom:** After what appeared to be a crash (browser returns `ERR_CONNECTION_REFUSED`), attempting to restart the server with `npm start` fails with: "Error: listen EADDRINUSE: address already in use :::3000".

**Diagnosis:** When a Git Bash terminal is closed without first stopping Node (Ctrl+C), the underlying Node process is not terminated. It continues running in the background, holding port 3000. Subsequent `npm start` attempts cannot bind to the port.

This is a Windows + Git Bash environment quirk, not a code defect — but it disrupts the development workflow and could be mitigated.

**Mitigation:** Use `Ctrl+C` to stop Node cleanly before closing the terminal. If the issue occurs, recover with `taskkill //F //IM node.exe`.

**Future improvement:** Add a clear "How to stop the server" section to the project README. Consider adding a graceful shutdown handler in `server.js` that releases the port on `SIGTERM`/`SIGINT`.

## 3. Good Security Features Observed

Despite the bugs above, the AI-generated base implementation includes several solid security design choices worth documenting. A case study should highlight what works, not only what fails.

### 3.1 Authentication

**JWT (JSON Web Tokens) with 2-hour expiry.** Voter and admin sessions are stateless and time-limited. Tokens are signed with a server-side secret. Expired tokens are rejected at the API layer.

**bcrypt password hashing with cost factor 12.** Passwords are never stored in plaintext. The cost factor of 12 (rather than the minimum of 10) provides strong resistance against offline cracking — at the cost of slightly slower login validation, which is appropriate for an election system.

**Forced admin password change on first login.** The default admin password (`admin123`) prompts an in-system change before granting full access. This mitigates the most common operational security mistake — forgetting to rotate default credentials.

### 3.2 Ballot Secrecy

**Salted voter hash, not voter ID.** A vote record stores `voter_hash` — a salted SHA-256 hash of (voter ID + election ID) — instead of the voter ID directly. This means even an administrator with full database access cannot trivially link a vote to a voter.

The salt is held server-side (in environment variable `VOTE_SALT`). Without the salt, the hash cannot be reverse-engineered to identify the voter, but the system can still verify that a given voter has voted.

### Ballot secrecy: the audit log can re-link voter to vote

The `votes` table deliberately stores a salted `voter_hash` rather than a voter
id, so that reading the votes table alone does not reveal who cast which ballot.
This is the intended ballot-secrecy property and it holds against an outsider
reading that one table.

It does **not** hold once the audit log is considered alongside it. Every cast
writes a `VOTE_CAST` row to `audit_log` keyed by the real `voter_id` with a
timestamp, while the `votes` table records rows in insertion order (`id`) each
with its own `created_at`. Because both orderings derive from the same sequence
of casts, the ordered `VOTE_CAST` entries can be correlated with the ordered
`votes` rows to re-link a voter to their specific vote — **without needing the
`VOTE_SALT` at all.** This is a sharper secrecy leak than the salt-knowledge case,
because it requires no secret, only read access to both tables.

The fix is a design change, not a parameter: do not record per-voter vote events
in a separately-ordered log that can be aligned with the votes table. Either omit
the per-voter `VOTE_CAST` timestamp, decouple vote insertion order from cast order,
or record the audit event without the ordering signal that enables the correlation.

This observation only surfaced from reading `server.js` (the cast path and audit
call) and `db.js` (the two tables' structures) together — neither file shows it
alone. It is cross-referenced in SECURITY-ANALYSIS.md §5.

### 3.3 Vote Integrity

**SHA-256 hash chain across all votes.** Every vote includes a `prev_hash` (the previous vote's hash) and a `vote_hash` (this vote's hash). Together they form a chain — modifying any historical vote breaks the chain, and the break can be detected.

**Integrity check endpoint** (`POST /api/admin/integrity`). An administrator can run a check that recomputes the entire chain and verifies that no votes have been tampered with. The system reports "Vote chain is intact — no tampering detected" if the verification passes.

This is the same cryptographic pattern used by Bitcoin's blockchain, applied to ballot integrity. It does not prevent tampering, but it makes tampering detectable.

### 3.4 Public Verifiability

**Receipt-based vote verification.** Each cast vote produces a unique receipt code (e.g., `A0C4D8773A70311E7E`). Anyone with this receipt can verify on the public results page that the vote exists in the system — **without** revealing which candidate received the vote.

This combines two normally opposing properties: public verifiability and ballot secrecy.

### 3.5 Audit Trail

**Comprehensive audit log.** Every significant action is recorded: admin logins, voter logins, registrations, vote casts, system seeds, integrity checks. Each entry includes timestamp, actor, action, details, and source IP.

This provides forensic traceability — essential for resolving election disputes or detecting unusual activity.

### 3.6 Operational Hardening

**Rate limiting.** Login attempts are limited to 50 per 15-minute window per IP; vote attempts to 10 per minute per IP. This slows brute-force attacks and reduces spam risk.

**Helmet middleware.** Standard HTTP security headers are set automatically (CSP, X-Frame-Options, X-Content-Type-Options, etc.), mitigating common web vulnerabilities like clickjacking and MIME sniffing.

**Parameterised SQL queries.** All database queries use prepared statements (`db.prepare(...).run(...)`), preventing SQL injection.

These features represent thoughtful security architecture for a demonstration system. For production use in a real election, several additional measures would be required (see SECURITY-ANALYSIS.md, future work).

## 4. User Experience (UX) Observations

Beyond functional bugs, several user-experience patterns are worth documenting. UX matters in election software because **voters and administrators must trust what they're using** — and trust is built (or broken) by clarity.

### 4.1 Demo credentials displayed on the login page

**Observation:** The login page footer prints the default voter and admin credentials in plain text: "Demo accounts — Voter: VIN0000001 / voter123 • Admin: admin / admin123"

**Strength:** Excellent for a demo or evaluation environment. Reviewers do not need to ask "how do I log in?" — the credentials are visible.

**Weakness:** Disastrous in production. Any deployment that doesn't remove this footer leaks default credentials publicly. The lesson: **demo-friendly defaults are a liability if they reach production.**

For the Ogun State case study, the footer is left in place because this remains a demo. A real deployment must remove or conditionally hide it (e.g., based on `NODE_ENV !== 'production'`).

### 4.2 No visible receipt after voting

**Observation:** When a voter casts a vote, the page transitions to a "You have voted" state, but the actual receipt code is never shown.

**Impact:** The voter cannot independently verify their vote was recorded. The receipt verification feature exists, but the voter has nothing to verify.

This is logged as Bug 2 in Section 2. UX-wise, it represents a missed opportunity to build voter confidence at the critical moment immediately after a ballot is cast.

### 4.3 Browser-native optgroup styling has platform limits

**Observation:** The LGA dropdown groups options by senatorial district using HTML `<optgroup>`. CSS attempts to make the district headings bold, italic, and coloured green for visibility.

**Result on Chrome / Windows:** Bold and colour applied; italic did not. Native OS dropdown controls override custom font styles.

**Implication:** For a polished production UI, a custom dropdown component (e.g., Select2, Choices.js) would be required to fully control appearance across all platforms. For a case study, the partial styling is sufficient — the grouping is visible.

### 4.4 "Sign in" button silently fails on CSP error

Related to Bug 1 in Section 2. From a UX perspective: the worst kind of button is one that produces no feedback when clicked. In our investigation, this caused several wasted diagnostic rounds before opening DevTools revealed the CSP block.

**Lesson:** Production election systems should display a clear error message for ANY login attempt — never a silent no-op. A voter who clicks "Sign in" and sees nothing happen has no path forward.

### 4.5 State field could not be defaulted in original code

The Ogun State adaptation changed the State field from free text (`<input>`) to a defaulted readonly field (`value="Ogun" readonly`). The original AI-generated code accepted any state value — which is a realistic concern for a system intended to model voter registration within a single state.

This is documented further in OGUN-ADAPTATIONS.md (future work).

## 5. Lessons for the Ogun State Case Study

The bugs, security observations, and UX patterns above are not merely academic. They point to specific lessons relevant to anyone considering e-voting infrastructure for a Nigerian state — whether Ogun, Lagos, or any of the 36.

### 5.1 Validation must be backend, not just frontend

The original system allowed any text as an LGA on voter registration. The Ogun adaptation (Layer 2.2) enforces validation at the API layer against a seeded `lgas` table — independent of any frontend dropdown.

**Why this matters:** A dropdown in the UI is a convenience for honest users. It does nothing to prevent an attacker bypassing the UI with a direct API call. **Validation belongs at the backend.** The frontend dropdown (Layer 2.3) is for UX, not security.

This is a general principle: **never trust client input.** Every API endpoint must independently validate every field, regardless of what the UI does.

### 5.2 Real Nigerian electoral structure maps cleanly to data

The Ogun State case study revealed that Nigeria's electoral structure — states → senatorial districts → LGAs → polling units — is hierarchical and well-suited to a relational database design.

The `lgas` table introduced in Layer 2.1 stores both `name` and `senatorial_district`. This single column enables grouping (UI), validation (backend), and future aggregation by district. The same pattern could be extended to polling units in a Layer 3 expansion.

### 5.3 Cryptographic integrity is necessary but not sufficient

The base system's hash-chain integrity and receipt verification are mathematically sound. They demonstrate that once a vote is recorded, it cannot be tampered with undetected, and any observer with a receipt can confirm the vote exists.

**However:** the cryptography assumes the voter has correctly identified themselves at login. If a voter's PVC and password are compromised — shared, stolen, or exposed in a breach — the system will faithfully record the impersonator's vote as legitimate. The cryptography protects what's recorded; it doesn't authenticate the person.

**This case study explicitly scopes voter authentication as out of scope.** Solving secure voter authentication for remote e-voting is a separate, equally important problem with its own active research landscape — including national digital ID systems, supervised in-person digital voting, biometric verification, and trusted-device attestation. Each approach trades off accessibility, security, and cost differently.

What this case study demonstrates is the **integrity and records layer** that any of those authentication approaches would still need: the cryptographic chain, the audit log, the receipt system, the validated geographic data. A complete e-voting deployment would combine this integrity layer with an authentication layer suited to the specific country, legal framework, and risk model.

### 5.4 Operational concerns are as important as code

Of the three bugs documented in Section 2, only one (CSP) was a code defect. The other two — receipt display and zombie processes — are operational gaps: how the system is run, deployed, monitored.

For an actual election deployment, the largest risks are unlikely to be in the cryptographic core. They are more likely in:

- Database backup and recovery procedures
- Server monitoring and uptime
- Incident response runbooks
- Staff training on operating the admin dashboard
- Legal and audit trail retention policies

These are not visible in the codebase but are essential to running any real election.

### 5.5 AI-generated code is a starting point, not an endpoint

The base implementation produced a working, security-conscious system in approximately 24 minutes of AI generation time. That is genuinely impressive.

But studying it carefully revealed:

- An internally inconsistent CSP configuration (Bug 1)
- A missing UI flow for showing receipts to voters (Bug 2)
- No backend validation of geographic fields like LGA

These are not failures of the AI tool — they are normal gaps in any first-pass code. The lesson is that **AI-generated code requires the same review, testing, and adaptation as any other code.** Treating it as finished, deployable software would be a mistake.

For this case study, the adaptations made in Layers 1, 2.1, 2.2, and 2.3 transform the generic system into a more honest representation of what Ogun State-specific e-voting would actually require.
